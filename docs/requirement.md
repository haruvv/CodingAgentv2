# AI自律開発エージェントチーム（MVP）要件定義書

本書は、Claude Code および GitHub Actions を活用し、ハーネスエンジニアリングの考え方に基づく自律開発パイプラインのMVPを実現するための要件を定義するものである。

なお、本書において以下の用語を用いる。

- **本MVP**：本書で定義する自律開発パイプラインそのもの
- **開発対象**：本MVPが開発を進めるアプリケーション、サービス、または関連成果物
- **自律開発基盤**：本MVPを構成するワークフロー、運用ルール、設定、および実行基盤

---

## 1. プロジェクト概要

### 1.1 目的
本MVPの目的は、AIエージェントが開発対象の開発を継続的かつ安定的に遂行できる実行環境を整備し、**「人間がチャットで自然言語の要求を送った場合に、一定条件のもとでマージ可能な Pull Request まで自律的に到達できる状態」**を最小構成で実現することにある。

人間と自律開発パイプラインとの接点は、GitHub Issues の直接起票ではなく、Discord ボット（agent-intake）を介したチャットとする。人間は自然言語で要求を入力し、agent-intake がその内容を解釈して GitHub Issue を起票することで、以降の自律開発パイプラインが起動する。

### 1.2 基本方針
本MVPは、以下の方針に基づいて構築する。

- **手作業による実装を原則としないこと**  
  本MVPが管理対象とするリポジトリにおいて、リポジトリ構造、CI設定、開発対象のコード、関連ドキュメントを含め、可能な限りエージェントによる生成を前提とする。

- **リポジトリを記録システムとして運用すること**  
  開発対象に関する要件、設計、実行計画、意思決定ログ等を `docs/` 配下に蓄積し、エージェントが参照可能な一次情報として管理する。

- **役割分担されたエージェントによる段階的処理を行うこと**  
  開発対象の進行に必要な要件整理、設計、タスク分割、実装、レビュー、監視等の役割を分離し、工程ごとに段階的に処理を進行させる。

- **人間は環境整備および意思決定を担うこと**  
  人間は主として、自律開発基盤を整備し、必要に応じて意思決定およびエスカレーション対応を行う。

## 2. 自律開発パイプラインの処理工程

本章で示す工程は、本MVP自体の実装工程ではなく、本MVPが開発対象の開発を進行する際の処理工程を示すものである。  
本MVPは、開発対象の進行にあたり、以下の工程を段階的に処理する。

### 2.0 要求受付（agent-intake）
Discord のスラッシュコマンド（`/issue`）を通じて、人間が自然言語で要求を入力する。agent-intake がその内容を LLM で解釈し、要求が曖昧な場合は Discord 上で確認を求める。要求が明確な場合は、GitHub Issue を自動起票して後続工程に引き渡す。

### 2.1 要求・要件定義
Issue 上の対話を通じて、要求内容の曖昧さを解消し、要件を確定する。

### 2.2 設計
確定した要件に基づき設計書を作成し、実装方針、制約条件、作業分割方針を明確化する。

### 2.3 タスク分割
設計内容をもとに、実装可能な粒度まで作業単位を分割する。

### 2.4 実装
エージェントが作業単位に従い、実装、テスト、関連ドキュメント更新、および Pull Request 作成を実施する。

### 2.5 レビュー・修正
レビュー結果、CI 結果、および実行結果に基づき、必要な修正を反復的に実施する。

### 2.6 統合
所定の条件を満たした成果物のみを統合対象とする。

## 3. エージェント構成

本MVPでは、開発対象の進行に必要な役割として、以下のエージェント群を実装対象とする。

| エージェント種別 | 主な役割 | 次工程 |
|---|---|---|
| 要件定義エージェント | Issue 上での要件整理および要求明確化 | 設計エージェント |
| 設計エージェント | 設計書作成、実装方針整理、タスク分割方針策定 | タスク分割エージェント |
| タスク分割エージェント | 実装可能な粒度への作業分割 | 実装エージェント |
| 実装エージェント | 実装、テスト、ドキュメント更新、Pull Request 作成 | レビューエージェント |
| レビューエージェント | 指摘抽出、修正要否判断、修正ループ制御 | 統合判定エージェント |
| 統合判定エージェント | レビュー結果および検証結果に基づく統合可否判定 | クローズまたは人間レビュー |
| 監視エージェント | 停滞検知、自動回復、必要時のエスカレーション | 再試行または人間対応 |

## 4. 実行基盤要件

### 4.1 知識構造の管理
- エージェントが参照先を把握するための簡潔な案内文書を用意すること。
- 当該案内文書は、詳細マニュアルではなく、管理対象リポジトリの構造および主要な参照先を示す入口として維持すること。
- 開発対象に関する詳細な要件、設計、運用ルール、判断基準は `docs/` 配下に分割して配置すること。
- `docs/` は説明資料ではなく、エージェントが開発を進める際に参照する記録システムとして扱うこと。
- ドキュメントには、最低限以下の情報を含めること。
  - 要件
  - 設計
  - 実行計画
  - 意思決定ログ
  - 運用ルール
  - 制約一覧

### 4.2 実行計画および意思決定ログ
- 開発対象または自律開発基盤に対する複雑な変更、または複数の Pull Request にまたがる変更に対しては、実装前に実行計画を作成すること。
- 実行計画には、少なくとも以下の内容を含めること。
  - 目的
  - 対象範囲
  - 前提条件
  - 実装方針
  - 検証方法
- 実装中に発生した重要な設計変更や判断については、意思決定ログとして記録すること。
- エージェントは、設計書に加え、実行計画および意思決定ログも参照して作業を行うこと。

### 4.3 決定論的ガードレール
- ツール実行後には、リンター、フォーマッター、型検査、テスト等の決定論的な検証を自動的に実施すること。
- エージェントの主観的判断に依存せず、同一条件下で再現可能な検証状態を維持すること。
- 失敗時には、その結果を記録し、次回以降の修正ループに利用可能とすること。

### 4.4 アーキテクチャ制約
- 開発対象におけるビジネスドメインごとの依存関係は、機械的に検証可能な形で管理すること。
- MVPにおいては、最低限以下の不変条件を定義対象とする。
  - レイヤ逆流の禁止
  - ドメイン外直接参照の禁止
  - 境界における型またはスキーマ検証の必須化
  - コード、設定、設計情報の配置ルールの明確化
- 制約違反は、レビュー以前に検出可能であること。

### 4.5 観測可能性
- エージェントが、自らの変更結果を検証できるよう、実行結果を機械的に参照できる状態を整備すること。
- MVPでは、少なくとも以下のいずれかを実現対象とする。
  - テスト結果の取得
  - アプリケーションログの取得
  - ビルド結果の取得
  - UI確認手段の確保
- 将来的には、ログ、メトリクス、トレース、UI操作結果等を自律的に確認可能な構成へ拡張できること。

### 4.6 停滞監視および自動回復
- 開発の実装またはレビューの進行が停滞した場合を検知できること。
- 停滞判定は単一条件によらず、進捗更新、成果物更新、応答状況、失敗反復等の複数シグナルを総合的に評価して行うこと。
- 停滞時には、再試行、処理再開、必要に応じた人間へのエスカレーションを実施可能であること。
- 自動回復には上限を設け、無制限な再試行を防止すること。
- 必要に応じて、ドキュメント、制約、検証手段等の不足を補うための改善タスクを起票可能であること。

### 4.7 CI/CD要件
- 最低限の検証として、本MVPを構成するワークフロー定義および設定ファイルの妥当性を確認可能であること。
- 開発対象の実装開始後は、段階的に以下を追加できること。
  - リント
  - 型検査
  - 単体テスト
  - 結合テスト
  - ビルド
  - データベース変更に関する検証
- パイプライン健全性を評価するため、成功率、失敗率、処理時間、再試行回数等を記録可能であること。

## 5. 統合条件

本MVPが生成した Pull Request の統合は、以下の条件をすべて満たした場合にのみ実施する。

- **レビュー通過**  
  所定のレビュー条件を満たしていること。

- **検証通過**  
  定義済みのビルド、テスト、リント等が成功していること。

- **明示的な統合許可**  
  自動統合を許可する明示的な条件または指示が存在すること。  
  当該条件または指示が存在しない場合は、人間による確認待ちとする。

## 6. リポジトリ運用方針

### 6.1 モノレポ構成
本MVPでは、エージェント定義、自律開発基盤に関する設計書、および開発対象本体を同一リポジトリに集約し、エージェントが必要なコンテキストへ到達しやすい状態を維持する。

### 6.2 ディレクトリ責務の分離
同一リポジトリ内に自律開発基盤と開発対象本体を共存させる場合であっても、責務の混在を防ぐため、ディレクトリ単位で役割を明確に分離すること。

### 6.3 ドキュメント配置方針
`docs/` 配下のドキュメントは、自律開発基盤に関する全体共通情報と、開発対象固有情報を分離して管理すること。  
これにより、開発対象本体と関連ドキュメントの対応関係を明確化し、エージェントによる対象範囲および参照先の誤認を防止する。

### 6.4 修正優先の運用
開発の進行においては、エージェントの処理能力を前提とし、長時間の待機よりも、小さな修正を繰り返しながら前進することを優先する。  
ただし、無制限な試行は許容せず、再試行上限およびエスカレーション条件を定義する。

### 6.5 定期的なガベージコレクション
- 開発対象および関連ドキュメントに対して、不要コード、重複ドキュメント、陳腐化したルール、技術的負債候補を定期的に検出可能であること。
- 検出結果に基づき、改善またはリファクタリングのためのタスクを起票可能であること。
- ドキュメントの鮮度および参照整合性についても定期点検の対象とする。

## 7. 非機能要件

### 7.1 再現性
同一入力に対し、可能な限り同一の検証結果および同一の実行手順が得られること。

### 7.2 可観測性
本MVPにおけるエージェントの失敗理由を、人間が事後的に追跡可能であること。

### 7.3 分割可能性
大きな要求を、安全に作業単位または複数の Pull Request に分割可能であること。

### 7.4 エスカレーション可能性
自律処理が困難な場合に、曖昧な状態で停止するのではなく、人間へ適切に引き渡し可能であること。

## 8. MVPの対象範囲

### 8.1 本MVPで目指すこと
- Discord チャットを起点として、要求受付から Pull Request 作成までの自律パイプラインを成立させること
- agent-intake が自然言語の要求を解釈し、GitHub Issue を自動起票できること
- Issue を起点として、要件整理から Pull Request 作成までの自律パイプラインを成立させること
- エージェントが参照可能な知識構造を `docs/` に整理すること
- 停滞時に検知、再試行、およびエスカレーションが可能であること
- 最小限の検証およびレビューによって、安全に前進可能であること

### 8.2 本MVPで対象外とすること
- 初期段階からの完全無人運用
- あらゆる種類の開発タスクへの対応
- 初回MVP時点における高度な監視基盤、完全自動UI検証、完全自動本番反映の実現
