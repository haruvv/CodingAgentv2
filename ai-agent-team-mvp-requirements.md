# AIエージェントチーム基盤 MVP 要件定義

## 1. 目的

GitHub を状態管理基盤、Claude を実行主体として利用し、  
**人間が作成した Issue をもとに、AI が実装作業を行い、Pull Request を作成する最小システム**を構築する。

本MVPの目的は、以下を検証することである。

- GitHub を中心とした状態遷移型の開発フローが成立すること
- Claude を実装担当エージェントとして運用できること
- 人間とAIの責任分界を明確にした上で、実装作業を半自動化できること
- 今後の複数エージェント化に向けた土台を作れること

## 2. スコープ

### 2.1 対象
本MVPで対象とするのは以下。

- GitHub Issue を起点とした開発開始
- GitHub Label による状態管理
- GitHub Actions を用いた Claude 実行トリガー
- Claude によるコード修正
- Claude によるコミットおよび Pull Request 作成
- CI 実行
- 人間による最終レビューとマージ判断

### 2.2 非対象
本MVPでは以下は対象外とする。

- 要件定義エージェント
- 設計専用エージェント
- QA専用エージェント
- 自動マージ
- 自動デプロイ
- Slack通知
- 複数エージェントの並列連携
- 本番監視や分析ダッシュボード
- 複雑な承認フロー

## 3. 想定ユーザー

### 3.1 オーナー / 人間
- Issue を起票する
- 完了条件を定義する
- AIが作成したPRをレビューする
- 最終的なマージ可否を判断する

### 3.2 Claude 実装エージェント
- Issue の内容を読み取る
- 指示に従ってコードを修正する
- 必要に応じてテストを追加・修正する
- Pull Request を作成する

### 3.3 GitHub Actions
- 特定ラベルを契機に Claude 実行処理を起動する
- CI を実行する
- 実行結果に応じて状態更新を行う

## 4. MVPの提供価値

このMVPで得たい価値は以下。

- 人間が「実装してください」と都度細かく指示しなくても、Issue 単位でAIに作業を流せる
- 実装の開始条件と完了条件が明文化される
- AIが勝手に動くのではなく、GitHub上の状態に従って動く
- 人間は要件記述と最終レビューに集中できる

## 5. 前提条件

- GitHub リポジトリが存在すること
- GitHub Actions が利用可能であること
- Claude をCLIまたはAPI経由で実行できること
- Claude が対象リポジトリを読み書きできる実行環境があること
- 最低限の CI（例: lint / test）が定義されていること

## 6. 機能要件

### 6.1 Issue起票機能
人間は GitHub Issue を作成し、AI に実装させたい作業内容を記述する。

Issueには最低限以下を記載する。

- 目的
- 実装内容
- 完了条件
- 制約事項
- 関連ファイル / 関連Issue（任意）

#### 受け入れ条件
- Issueテンプレートに従って記載できること
- 必須項目が明確であること

### 6.2 状態管理機能
Issue と PR は Label により状態管理される。

#### Issueの状態例
- `todo`
- `ready-for-impl`
- `in-progress`
- `review-required`
- `done`
- `blocked`

#### PRの状態例
- `ai-generated`
- `ready-for-review`
- `changes-requested`

#### 受け入れ条件
- 人間および Actions が状態を識別できること
- 状態に応じて次の処理が明確であること

### 6.3 AI実装起動機能
Issue に `ready-for-impl` ラベルが付与されたら、GitHub Actions が起動し、Claude 実装エージェントを実行する。

#### 入力
- Issue本文
- Issueタイトル
- 関連コードベース
- 必要に応じて補足ルールファイル

#### 出力
- 対応ブランチでのコード変更
- コミット
- Pull Request の作成

#### 受け入れ条件
- 指定ラベル付与で自動起動すること
- 同一Issueに対して二重起動しないこと
- 起動失敗時に判別できること

### 6.4 実装機能
Claude は Issue の内容に基づいて必要なコード修正を行う。

#### 実装範囲
- アプリケーションコード修正
- 関連テストの追加/修正
- 最低限のドキュメント修正（必要な場合のみ）

#### 制約
- Issueで指定されていない大規模リファクタは行わない
- 無関係なファイル変更を避ける
- 秘密情報は生成・出力しない

#### 受け入れ条件
- 変更内容がIssueのスコープ内に収まること
- 無関係な差分が最小限であること

### 6.5 Pull Request作成機能
Claude は実装完了後、Pull Request を作成する。

PRには最低限以下を含める。

- 変更概要
- 対応Issue
- 実施内容
- 懸念点 / 未対応事項（あれば）

#### 受け入れ条件
- PRが自動作成されること
- Issue と PR が紐づくこと
- PR本文がテンプレートに沿っていること

### 6.6 CI実行機能
PR作成後に CI を実行する。

最低限の対象は以下。

- lint
- test

必要に応じて型チェックも含む。

#### 受け入れ条件
- PR作成時にCIが自動実行されること
- 成否がPR上で確認できること

### 6.7 人間レビュー機能
人間は AI が作成した PR をレビューし、以下のいずれかを判断する。

- マージ
- 修正依頼
- 却下

#### 受け入れ条件
- 人間が最終判断者であること
- 修正依頼時に再度Claudeへ戻せる余地があること
  - ただしMVPでは手動再実行でも可

## 7. 非機能要件

### 7.1 再現性
- 同じIssueテンプレート・同じ手順で起動できること

### 7.2 可観測性
- どのIssueが
- どの状態にあり
- Claude実行が成功したか失敗したか  
を GitHub 上で追えること

### 7.3 安全性
- 自動マージしないこと
- main / protected branch へ直接pushしないこと
- AIの変更は必ずPR経由とすること

### 7.4 保守性
- 今後、要件整理エージェントやレビューエージェントを追加できる構成であること

## 8. 基本フロー

1. 人間が Issue を作成する
2. 人間が `ready-for-impl` ラベルを付与する
3. GitHub Actions が起動する
4. Claude が Issue を読み、ブランチを作成する
5. Claude がコード修正・テスト修正を行う
6. Claude がコミットし PR を作成する
7. CI が実行される
8. 人間がレビューする
9. 問題なければマージする
10. Issue を `done` にする

## 9. エラー時の扱い

### 9.1 Claude起動失敗
- Issueを `blocked` または失敗判定用ラベルにする
- 人間が原因確認する

### 9.2 実装失敗
- PR未作成のまま終了した場合は失敗扱いとする
- ログから確認できること

### 9.3 CI失敗
- PRは作成済みだがレビュー前に修正必要状態とする
- MVPでは人間が再実行判断してよい

### 9.4 Issue記述不足
- Claude が適切に処理できない場合、処理を中断できること
- 不足項目が分かるようにすること

## 10. MVP完了条件

以下を満たしたらMVP完成とする。

- Issueテンプレートが整備されている
- `ready-for-impl` ラベルで Actions が起動する
- Claude がコード修正を行い PR を作成できる
- CI が自動実行される
- 人間がPRレビューしてマージできる
- 1つ以上の実際のIssueで end-to-end に動作確認できる

## 11. 今後の拡張候補

MVP完了後の拡張候補。

- Issue分割エージェント
- レビュー専用エージェント
- Slack通知
- 自動再実行
- PRプレビュー環境
- 要件→設計→実装の多段エージェント化
- 自動スコアリングと改善提案
